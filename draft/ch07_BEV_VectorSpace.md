# 7장. BEV(Bird's-Eye View) 변환과 Vector Space

> **학습 목표**
> 이 장을 마치면 자율주행 인식이 왜 **탑뷰의 격자 공간**에서 표현되는지를 기하학적·공학적으로 설명할 수 있다. BEVFormer 계열의 Cross-Attention 기반 변환과 Tesla의 Vector Space 개념이 서로 다른 추상 수준에서 같은 문제를 어떻게 푸는지 이해하고, 이 세 표현(BEV, Occupancy, Vector Space)이 한 스택 안에서 어떻게 공존하며 서로 보완하는지를 그려낸다. PyTorch로 간단한 BEV 변환기를 직접 짜 보면서 개념을 손에 익힌다.

---

## 7.1 왜 인식 결과는 BEV에 놓여야 하는가

처음 자율주행에 진입한 사람은 보통 이런 직관을 갖는다. "카메라가 찍은 이미지에서 차량·차선·보행자를 바로 검출하면 되지 않나?" 이 직관은 시작점으로는 맞지만 곧 벽에 부딪힌다. 문제는 원근 공간의 기하학적 특성에 있다. 원근 이미지에서는 같은 크기의 자동차가 가까이 있으면 크게 보이고 멀리 있으면 작게 보인다. 두 카메라의 시야가 겹치는 영역에서는 하나의 자동차가 두 번 검출되고, 그 좌우 이미지에서의 위치는 서로 다르다. 무엇보다 **Planner가 "2m 옆으로 이동"이라는 결정을 내릴 때 그 2m가 이미지에서 몇 픽셀인지 계산해야 하는데**, 원근 이미지에서는 이 대응이 픽셀 위치에 따라 크게 달라진다.

이 불편을 해결하는 것이 BEV(Bird's-Eye View) 공간이다. 차량 위 몇 미터에서 아래를 내려다본 관점의 2D 평면이다. 이 공간에서는 모든 거리가 실제 미터 단위로 표현되고, 자동차·차선·보행자가 **원래 크기의 비율 그대로** 놓인다. 여러 카메라의 정보가 이 공간에서 자연스럽게 하나로 합쳐지고, Planner의 결정이 그대로 미터 단위의 궤적으로 연결된다. 한마디로, BEV는 **기하학적으로 말이 되는 공간**이다.

---

## 7.2 BEV 변환의 세 세대

BEV를 만드는 방법은 시간이 지나면서 크게 세 세대를 거쳤다.

1세대는 **역투시 변환(Inverse Perspective Mapping, IPM)** 이다. 지면이 평평하다는 가정 아래, 카메라의 내·외부 파라미터로 계산된 호모그래피를 써서 이미지를 위에서 내려다본 것처럼 변환한다. 이 방법은 단순하고 빠르지만, 가정이 자주 깨진다. 언덕이나 과속방지턱에서는 지면이 평평하지 않고, 장애물이 지면에서 솟아 올라 있으면 IPM 결과에서 **과장되게 커 보이는** 왜곡이 생긴다. 29장에서 MentorPi로 IPM을 실제로 돌려보면 이 한계가 얼마나 빨리 드러나는지 직접 체감할 수 있다.

2세대가 2020년 NVIDIA의 **Lift-Splat-Shoot(LSS)** 이다. 6장에서 Occupancy Network의 토대로 이미 다뤘다. 각 픽셀의 깊이 분포를 학습으로 추정해 3D 공간에 특징을 뿌리고, 그 3D 공간을 BEV 평면으로 투영한다. 언덕이나 솟은 장애물에 강건하고, 카메라 파라미터의 자연스러운 활용이 가능하다. 실무의 표준으로 빠르게 자리 잡았다.

3세대는 2022년의 **BEVFormer**다. LSS가 "이미지에서 BEV로 정보를 밀어넣는" 방향인 반면, BEVFormer는 반대로 "BEV에서 이미지를 끌어오는" 방향이다. BEV 격자의 각 셀을 **쿼리**로 만들고, 그 쿼리가 Cross-Attention을 통해 여러 카메라 이미지의 적절한 픽셀들을 **스스로 읽어온다**. 이 발상의 전환이 여러 측면에서 우월하다. 시야가 겹치는 영역에서 자연스럽게 정보 가중이 이루어지고, 새로운 카메라 구성이나 해상도 변화에도 유연하다. 2023~2024년에는 StreamPETR, MapTR 같은 계열이 등장하며 성능과 효율을 계속 개선하고 있다.

Tesla가 정확히 어떤 방식을 쓰는지는 공개되지 않았지만, 2022년 AI Day의 "학습된 위치 임베딩을 통한 다중 카메라 융합"이라는 표현은 BEVFormer 계열에 가까운 구조를 시사한다. 여러 카메라에서 동시에 정보를 받아 BEV 공간에서 융합하되, 그 융합 과정을 네트워크가 스스로 배우도록 한다는 철학은 동일하다.

---

## 7.3 Cross-Attention이 BEV에 들어온 방식

BEV 격자를 상상해 보자. 200×200 해상도의 격자를 100m × 100m의 실제 공간에 매핑하면, 각 셀은 0.5m 크기다. 각 셀에 대해 **쿼리 벡터** 하나씩을 둔다. 이 쿼리는 학습 가능한 파라미터에서 시작해, 점차 그 셀 위치에서 의미 있는 정보를 끌어오는 방향으로 조정된다.

이미지 쪽에는 각 카메라의 특징 맵이 있다. 특징 맵의 각 픽셀은 key와 value 벡터로 제공된다. 쿼리가 key와 내적을 통해 유사도를 계산하고, softmax로 정규화된 가중치로 value들을 가중 평균한다. 이 결과가 BEV 셀의 새로운 표현이 된다. 이것이 표준 Cross-Attention이고, Transformer의 핵심 메커니즘과 동일하다.

그러나 BEV 격자의 모든 셀이 모든 카메라 픽셀을 다 들여다보는 **Full Attention**은 계산량이 폭발한다. 200×200개 셀이 각각 여섯 개 카메라의 수만 픽셀을 참조한다면 현실적으로 실시간이 불가능하다. 실무의 해결책이 **Deformable Attention**이다. 각 쿼리는 미리 예측된 몇 개(보통 4~16개)의 주변 픽셀만 참조한다. 어떤 픽셀을 참조할지를 네트워크가 학습한다. 계산량이 수십 배 줄어들면서도 정확도는 오히려 올라간다. 모든 곳을 다 보는 것이 반드시 좋은 것은 아니라는 반직관적 결과를 BEVFormer가 깔끔하게 보여준다.

## 7.4 Vector Space — Tesla의 한 발 더

Tesla의 인식은 BEV에서 한 단계 더 나아간다. BEV가 여전히 격자 기반의 조밀한 표현인 반면, Tesla는 객체·차선·신호등·정지선 같은 **개별 실체(entity)** 를 명시적인 벡터로 다루는 **Vector Space** 개념을 도입한다. 각 실체는 위치·속도·분류·불확실성 같은 속성을 담은 하나의 벡터가 되고, 장면 전체는 이 벡터들의 **리스트**로 표현된다.

이 차이가 왜 중요한가. BEV는 공간의 기하를 조밀하게 유지하는 데 강하지만, Planner가 직접 쓰기에는 정보가 너무 분산되어 있다. Planner는 "옆 차량의 속도는 얼마, 그 차가 끼어들 확률은 얼마"처럼 **객체 단위의 추론**을 하고 싶어 한다. Vector Space는 그 추론에 바로 들어맞는 표현이다. BEV 격자에서 객체들을 **분리·요약**한 결과가 Vector Space라고 이해하면 직관적이다.

내부적으로는 Set Transformer나 Slot Attention 같은 메커니즘이 쓰인다. BEV 표현에서 N개의 "엔티티 쿼리"가 각자 장면의 한 객체를 담당하도록 학습된다. 이 구조는 객체 검출의 현대적 표준인 DETR 계열과 닮았다. DETR이 이미지에서 객체 쿼리를 뽑는다면, Tesla의 Vector Space는 BEV 공간에서 장면 엔티티 쿼리를 뽑는다.

## 7.5 세 표현의 공존 — 왜 하나로 통일하지 않는가

"BEV와 Occupancy와 Vector Space, 이 셋을 왜 다 갖고 있어야 하나?"가 공부하며 자주 떠오르는 의문이다. 답은 각 표현이 **다른 목적에 최적화**되어 있다는 것이다.

BEV는 조밀한 2D 표현으로, 차선과 주행 가능 영역 같은 **연속적·지면 기반** 정보를 다루기에 적합하다. Occupancy는 3D 복셀로 점유 여부를 나타내므로, **입체적 장애물과 미지의 물체**를 처리하는 데 강하다. Vector Space는 희소한 엔티티 리스트로, **Planner가 직접 추론할 재료**로 최적이다. 한 표현이 다른 표현의 장점을 모두 흡수하지 못한다. 그래서 세 표현이 한 스택 안에 공존하며, 필요에 따라 서로 다른 헤드가 참조한다.

스택의 그림을 그려 보면 이해가 빨라진다. 이미지 백본에서 나온 공통 특징이 먼저 BEV 공간으로 올라가고, 그 위에서 Occupancy 헤드는 3D 복셀 점유를 예측하고, Vector Space 헤드는 엔티티 리스트를 추출한다. Planner는 이 세 표현을 모두 입력으로 받아 궤적을 결정한다. 각 표현의 품질이 서로를 보완한다. 하나의 표현에서 놓친 정보가 다른 표현에서 잡혀 주는 구조다.

## 7.6 미니 BEV 변환기 — 학습 가능한 IPM

복잡한 BEVFormer까지 가기 전에, 학습 가능한 보정이 붙은 IPM을 먼저 만들어 보는 것이 교육적이다. 아래 코드는 단일 카메라에서 IPM을 기반으로 BEV를 만들되, 네트워크가 그 결과를 미세 조정할 수 있도록 한다.

```python
import torch
import torch.nn as nn
import torch.nn.functional as F

class LearnedIPM(nn.Module):
    def __init__(self, H_init, bev_shape=(200, 200)):
        super().__init__()
        self.H = nn.Parameter(H_init.clone())
        self.bev_h, self.bev_w = bev_shape
        self.correction = nn.Sequential(
            nn.Conv2d(3, 16, 3, padding=1), nn.SiLU(),
            nn.Conv2d(16, 3, 3, padding=1),
        )

    def forward(self, img):
        B = img.size(0)
        ys, xs = torch.meshgrid(
            torch.linspace(-1, 1, self.bev_h, device=img.device),
            torch.linspace(-1, 1, self.bev_w, device=img.device),
            indexing='ij')
        ones = torch.ones_like(xs)
        grid_bev = torch.stack([xs, ys, ones], dim=-1)
        pts = grid_bev @ torch.linalg.inv(self.H).T
        pts = pts[..., :2] / pts[..., 2:3].clamp(min=1e-6)
        grid = pts.unsqueeze(0).expand(B, -1, -1, -1)
        bev = F.grid_sample(img, grid, align_corners=False)
        bev = bev + self.correction(bev)
        return bev
```

이 구현의 H는 초기 호모그래피 추정값에서 시작해 학습으로 조정된다. 그 결과를 작은 CNN이 더 미세하게 보정한다. 지면이 완전히 평평하지 않은 상황에서도 어느 정도 견디는 실용적인 출발점이 된다. 24·25장에서 이보다 발전된 구조로 확장한다.

## 장말 정리

BEV는 자율주행 인식이 Planner를 제대로 서빙하기 위해 반드시 거쳐야 하는 공간이다. IPM에서 Lift-Splat으로, 다시 BEVFormer로 발전한 흐름은 모두 "어떻게 여러 카메라의 정보를 탑뷰 격자로 올바르게 융합할 것인가"의 답을 찾아온 과정이다. Tesla는 이 BEV 위에 한 층 더 올려 개별 엔티티의 벡터 표현인 **Vector Space**를 만든다. BEV·Occupancy·Vector Space의 세 표현이 한 스택 안에서 공존하며, 각자의 목적에 맞는 역할을 분담한다.

3. Vector Space 표현이 Planner 입력으로 유리한 이유 세 가지를 구체적 예와 함께 설명하라.
